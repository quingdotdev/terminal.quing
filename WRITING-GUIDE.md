# writing guide

this guide governs docs, release notes, and in app copy. it aligns with the Quing brand rules and the traditional, direct voice used across the project.

## voice

write with clarity and restraint. avoid hype and avoid sycophancy. prefer short sentences and plain language. do not use emojis and do not use em dashes.

## casing and names

Quing is always title case. slate is always lowercase. headings are lowercase except for required brand names.

## structure

prefer prose. use lists only when a table or paragraph would be unclear. do not use lists of three or five items.

## formatting rules

| rule | guidance |
| --- | --- |
| links | keep them minimal and relevant |
| code | use fenced code blocks with language tags |
| commands | show exact commands and keep them copy ready |
| dates | use full dates for clarity |
| tone | direct, measured, traditional |
| scope | focus on user impact, not internal chatter |

## terminology

use product terms consistently. a project contains tabs. a tab contains panes. a pane is backed by a pty process. a profile defines shell, args, and cwd.
